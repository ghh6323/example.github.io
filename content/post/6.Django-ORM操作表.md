## 〇、先用一个比喻搞懂 ORM

假设你不懂英语（SQL），但你要跟一个只懂英语的外国仓库管理员（数据库）打交道，怎么办？—— 请一个**翻译**。

这个翻译就是 **ORM（Object-Relational Mapping，对象关系映射）**。

```
你用中文说："帮我把最贵的 3 件商品找出来"
        ↓  ORM 翻译
数据库收到：SELECT * FROM home_codeproduct ORDER BY price DESC LIMIT 3;
        ↓
数据库把结果给 ORM，ORM 包装成 Python 对象还给你
```

**对照关系（背下来）：**

| Python 世界 | 数据库世界 | 类比 |
|---|---|---|
| 一个类（class） | 一张表（table） | "商品"这个模板 |
| 类的属性 | 表的字段/列（column） | 商品名、价格、库存 |
| 一个对象 | 表里的一行（row） | 某一件具体商品 |

**用 ORM 的两大好处：**

1. **全程写 Python，不用写 SQL** —— 对新手友好
2. **换数据库零成本** —— 开发用 SQLite，上线换 MySQL，业务代码一行不用改，只改 settings.py 里的连接配置

## 二、创建表结构：三步曲

### 第 1 步：在 models.py 里"画图纸"

打开 `home/models.py`，写一个类：

```python
from django.db import models


class CodeProduct(models.Model):
    """代码商品表"""
    title = models.CharField('商品名', max_length=100)
    description = models.TextField('商品描述', blank=True)
    price = models.DecimalField('价格', max_digits=8, decimal_places=2)
    stock = models.IntegerField('库存', default=0)
    is_on_sale = models.BooleanField('是否上架', default=True)
    created_at = models.DateTimeField('创建时间', auto_now_add=True)
    updated_at = models.DateTimeField('更新时间', auto_now=True)

    def __str__(self):
        return self.title
```

逐行用大白话解释：

| 代码 | 人话解释 |
|---|---|
| `class CodeProduct(models.Model)` | 定义一张表。必须继承 `models.Model`，相当于告诉 Django"我是一张表，归你管" |
| `CharField('商品名', max_length=100)` | 短文本列。`max_length` 是**必填**的（数据库要知道给这列留多大空间）。第一个参数'商品名'是中文标签，以后后台管理界面会显示它 |
| `TextField('商品描述', blank=True)` | 长文本列（不限长度）。`blank=True` 表示表单里可以不填 |
| `DecimalField(max_digits=8, decimal_places=2)` | 精确小数列，存钱专用。`max_digits=8` 一共最多 8 位数字，`decimal_places=2` 其中小数点后 2 位 → 最大能存 999999.99。**存金额永远用 DecimalField，不要用 FloatField**（浮点数有精度误差，0.1+0.2≠0.3 那种坑） |
| `IntegerField(default=0)` | 整数列，默认 0 |
| `BooleanField(default=True)` | 是非列，只有 True/False 两种值 |
| `auto_now_add=True` | **创建那一刻**自动填入当前时间，之后不再变（适合"创建时间"） |
| `auto_now=True` | **每次保存**都自动刷新成当前时间（适合"更新时间"） |
| `def __str__(self)` | 定义这个对象"显示出来长什么样"。不写的话后台和 shell 里显示的是 `CodeProduct object (1)` 这种看不懂的东西；写了就显示商品名 |

### 常用字段类型速查表

| 字段 | 装什么 | 必记参数 | 生活例子 |
|---|---|---|---|
| `CharField` | 短文本 | `max_length`（必填） | 标题、姓名、手机号 |
| `TextField` | 长文本 | `blank=True` | 文章正文、商品描述 |
| `IntegerField` | 整数 | `default=0` | 库存、年龄、点赞数 |
| `DecimalField` | 精确小数 | `max_digits` + `decimal_places` | 价格、余额 |
| `FloatField` | 浮点数 | — | 评分（不精确也无所谓时用） |
| `BooleanField` | 是/否 | `default=True` | 是否上架、是否已读 |
| `DateTimeField` | 日期+时间 | `auto_now_add` / `auto_now` | 创建时间、登录时间 |
| `DateField` | 只要日期 | — | 生日 |
| `EmailField` | 邮箱 | — | 注册邮箱（自动校验格式） |
| `URLField` | 网址 | — | 商品链接 |
| `ForeignKey` | 一对多关系 | `on_delete=models.CASCADE`（必填） | 一个用户有多个订单 |
| `ManyToManyField` | 多对多关系 | — | 一篇商品有多个标签 |

### 两个最容易混的参数：null 和 blank

```python
# null 和 blank 不是一回事！
description = models.TextField(blank=True)              # 推荐：字符串字段只加 blank
nickname = models.CharField(max_length=50, null=True, blank=True)  # 也可以两个都加
```

- **`null=True`**：跟**数据库**说——这列允许存 NULL（空值）
- **`blank=True`**：跟**表单校验**说——用户提交表单时这项可以不填

**经验法则**：字符串类字段（Char/Text）一般**只加 `blank=True` 不加 `null=True`**。否则一个"空"会有两种形态（NULL 和 `''`），查询时要判断两次，自找麻烦。数字、日期字段想可空就得加 `null=True`。

### 第 2 步：makemigrations —— 生成"施工图纸"

```bash
python manage.py makemigrations
```

输出类似：

```
Migrations for 'home':
  home/migrations/0001_initial.py
    - Create model CodeProduct
```

**这一步干了什么**：Django 拿着你的 models.py，和数据库当前的样子做对比，把"差异"写成一份 Python 文件，存在 `home/migrations/0001_initial.py`。

⚠️ **注意：这一步还没有动数据库！** 它只是写好了"施工图纸"。

> 💡 好奇宝宝可以偷看一下图纸内容：`type home\migrations\0001_initial.py`（Windows），里面就是一份变更清单。
> 想看它最终会执行什么 SQL？`python manage.py sqlmigrate home 0001`

### 第 3 步：migrate —— 真正"施工"

```bash
python manage.py migrate
```

输出：

```
Applying home.0001_initial... OK
```

现在数据库里真的有了 `home_codeproduct` 这张表。

**两个冷知识：**

1. **表名规则**：`app名_类名小写`。`home` app 的 `CodeProduct` → `home_codeproduct`。你 SSO 项目里的用户表叫 `auth_user`，就是 `auth` app 的 `User` 类
2. **id 主键是白送的**：你没有定义 id，但 Django 自动给每张表加了自增主键 `id`（1、2、3……自动往上加）

### 三步曲口诀

```
改 models.py（画图纸） → makemigrations（写变更单） → migrate（施工）
```

以后不管是建表还是改表，**永远走这三步**，缺一不可。

## 三、修改表结构

**核心心法：永远不直接动数据库，只改 models.py，然后跑三步曲的后两步。**

### 场景 1：加字段（最常见）

models.py 里加一行：

```python
class CodeProduct(models.Model):
    # ...原有字段保持不动...
    category = models.CharField('分类', max_length=50, default='未分类')   # 新增
```

然后：

```bash
python manage.py makemigrations    # 生成 0002_codeproduct_category.py
python manage.py migrate
```

完事。**已有的数据不会丢**，老数据的新字段统一填默认值'未分类'。

### ⚠️ 新手 100% 会遇到的拦截提问

如果你加字段时**忘了写 `default`**，makemigrations 会拦住你：

```
You are trying to add a non-nullable field 'category' to codeproduct
without a default; we can't do that (the database needs something to
populate existing rows).
Please select a fix:
 1) Provide a one-off default now (will be set on all existing rows)
 2) Quit, and let me add a default in models.py
Select an option:
```

**它在说什么**："表里已经有数据了，你新加的这列不允许为空，那老数据的这列填什么？你得告诉我。"

**两个选项怎么选**：

- 选 `1`：现场输一个临时默认值（比如输 `'未分类'`），只用于这一次填充老数据
- 选 `2`：先退出，回到 models.py 给字段加上 `default=...`（或 `null=True`），再重新跑 makemigrations ← **推荐，一劳永逸**

### 场景 2：改字段

想把商品名长度从 100 改成 200？**直接改 models.py 里的那行代码**：

```python
title = models.CharField('商品名', max_length=200)   # 100 → 200
```

然后三步曲后两步。Django 自动生成 `ALTER TABLE` 语句，**数据原封不动**。

⚠️ 注意：`max_length` **改大安全，改小危险**——如果有数据超过了新长度，可能被截断。

### 场景 3：改字段名

```python
title = models.CharField('商品名', max_length=100)
# 改成
name = models.CharField('商品名', max_length=100)
```

跑 makemigrations 时 Django 会问：

```
Did you rename codeproduct.title to codeproduct.name (a CharField)? [y/N]
```

输入 `y`，它就知道你是**改名**而不是"删了旧的建新的"——数据得以保留。如果不小心选了 N，它会当成删旧列建新列，**那一列的数据就没了**。

### 场景 4：删字段

把 models.py 里那行**删掉**（或注释掉），跑三步曲后两步。

⚠️ **该列数据会永久删除**，删之前想清楚。

### 迁移文件的黄金纪律

1. **改一点就迁移一次**，别攒一堆改动一起迁，出问题好定位
2. `migrations/` 文件夹是**历史档案**，不要手动改、不要随便删里面的文件
3. 用 Git 时 `migrations/` 要**一起提交**——别人拉了你的代码，跑一遍 migrate 就能得到和你一样的表结构
4. **开发阶段**玩坏了想推倒重来：删掉 `db.sqlite3` 文件 + 删掉各 app `migrations/` 里除 `__init__.py` 外的文件 → 重新 `makemigrations` + `migrate`
5. **生产环境（线上）绝对不要**用第 4 条的办法——数据就是命

## 四、增删改查（CRUD）—— 本文主菜

打开练习场：

```bash
python manage.py shell
```

```python
>>> from home.models import CodeProduct
```

**所有数据库操作都通过一个"管家"：`模型类.objects`**。它的大名叫 Manager（管理器），增删改查全找它。

### 4.1 增（Create）

```python
# 方式一：create() —— 创建+保存一步到位（最常用）
>>> p = CodeProduct.objects.create(title='Django 实战源码', price=99.00, stock=10)
>>> p.id        # 保存后自动有了 id
1

# 方式二：先 new 一个对象，再 save()
>>> p = CodeProduct(title='YOLOv8 项目源码', price=199.00)
>>> p.stock = 5
>>> p.id        # 注意！现在还是 None，因为还没进数据库
>>> p.save()    # 这一行才真正写入数据库
>>> p.id
2

# 方式三：bulk_create 批量创建（一次 SQL 插多条，效率高）
>>> CodeProduct.objects.bulk_create([
...     CodeProduct(title='Vue 商城源码', price=88.00, stock=3),
...     CodeProduct(title='Flask 博客源码', price=66.00, stock=8),
... ])
```

**⚠️ 新手第一大坑**：方式二忘了 `save()`，对象只活在内存里，程序一关就没了。记住：

> **create() 自带保存；自己 new 的对象必须手动 save()。**

### 4.2 查（Read）—— 花样最多，重点学

#### ① 查全部

```python
>>> CodeProduct.objects.all()
<QuerySet [<CodeProduct: Django 实战源码>, <CodeProduct: YOLOv8 项目源码>, ...]>
```

返回的是 **QuerySet**——可以理解成"一个装着对象的列表"，能用 for 循环、能取下标。

#### ② 条件查询：filter（返回列表）

```python
# 单条件
>>> CodeProduct.objects.filter(category='未分类')

# 双下划线条件（重点！）
>>> CodeProduct.objects.filter(price__lt=100)           # 价格 < 100
>>> CodeProduct.objects.filter(title__contains='源码')   # 标题包含"源码"
>>> CodeProduct.objects.filter(stock__gte=5)            # 库存 >= 5

# 多条件 = 并且（AND）
>>> CodeProduct.objects.filter(price__gte=50, stock__gt=0)   # 价格>=50 且 有库存

# 查不到时返回空列表，不报错
>>> CodeProduct.objects.filter(title='不存在的商品')
<QuerySet []>
```

#### ③ 查单个：get（返回一个对象，不是列表）

```python
>>> p = CodeProduct.objects.get(id=1)
>>> p.title
'Django 实战源码'
>>> p.price
Decimal('99.00')
```

**⚠️ get 的两个雷区（考试高频）：**

```python
# 雷区一：查不到 → 抛 DoesNotExist 异常，页面直接 500
>>> CodeProduct.objects.get(id=999)
home.models.CodeProduct.DoesNotExist: CodeProduct matching query does not exist.

# 雷区二：查到多条 → 抛 MultipleObjectsReturned
>>> CodeProduct.objects.get(category='未分类')   # 如果"未分类"有好几个
home.models.CodeProduct.MultipleObjectsReturned: get() returned more than one CodeProduct

# 安全写法一：try/except
>>> try:
...     p = CodeProduct.objects.get(id=999)
... except CodeProduct.DoesNotExist:
...     p = None

# 安全写法二：filter + first（查不到返回 None，永不报错）
>>> p = CodeProduct.objects.filter(id=999).first()
>>> print(p)
None
```

**filter 和 get 的选择口诀：**

| | `filter()` | `get()` |
|---|---|---|
| 返回值 | QuerySet（列表） | 一个对象 |
| 查不到 | 空列表 `[]`，**不报错** | **抛异常** |
| 查到多个 | 都在列表里 | **抛异常** |
| 什么时候用 | 绝大多数情况 | 确定最多只有一条时（如按 id 查） |
| 保命写法 | `.first()` 取第一个或 None | try/except |

#### ④ 双下划线 `__` 查询语法速查（filter 的灵魂）

格式：`字段名__条件=值`（注意是**两个**下划线）

| 写法 | 含义 | SQL 对应 |
|---|---|---|
| `price__lt=100` | 小于 | `< 100` |
| `price__lte=100` | 小于等于 | `<= 100` |
| `price__gt=100` | 大于 | `> 100` |
| `price__gte=100` | 大于等于 | `>= 100` |
| `title__contains='源码'` | 包含（区分大小写） | `LIKE '%源码%'` |
| `title__icontains='django'` | 包含（**不**区分大小写） | 搜英文时用 |
| `title__startswith='Django'` | 以…开头 | `LIKE 'Django%'` |
| `title__exact='源码'` | 精确等于 | `= '源码'` |
| `id__in=[1, 2, 3]` | 在列表中 | `IN (1,2,3)` |
| `stock__isnull=True` | 是空值 | `IS NULL` |
| `created_at__year=2026` | 按年份 | — |
| `created_at__date=...` | 只按日期部分 | — |
| `price__range=(50, 100)` | 区间内 | `BETWEEN 50 AND 100` |

#### ⑤ 排除、排序、限量、统计

```python
# 排除：exclude 是 filter 的反义词
>>> CodeProduct.objects.exclude(is_on_sale=False)    # 排除所有下架的

# 排序：order_by，字段名前加负号表示倒序
>>> CodeProduct.objects.order_by('price')        # 便宜 → 贵
>>> CodeProduct.objects.order_by('-price')       # 贵 → 便宜
>>> CodeProduct.objects.order_by('-price', 'stock')  # 先按价格倒序，价格相同按库存正序

# 切片取前 N 条（对应 SQL 的 LIMIT）
>>> CodeProduct.objects.order_by('-price')[:3]   # 最贵的 3 个
>>> CodeProduct.objects.all()[5:10]              # 第 6~10 条（分页的原理）

# 只要部分字段：values 返回字典列表，values_list 返回元组列表
>>> CodeProduct.objects.values('title', 'price')
<QuerySet [{'title': 'Django 实战源码', 'price': Decimal('99.00')}, ...]>

# 统计
>>> CodeProduct.objects.count()                      # 总数
>>> CodeProduct.objects.filter(stock=0).exists()     # 是否存在（比 count 快）
```

#### ⑥ 链式调用：像搭积木一样组合

```python
# filter → exclude → order_by → 切片，一路串下去
>>> CodeProduct.objects.filter(is_on_sale=True).exclude(stock=0).order_by('-created_at')[:5]
# 人话：上架的、排除没货的、按上架时间倒序、取前 5 个
```

#### ⑦ 进阶：OR 条件用 Q 对象

filter 的多条件天然是 AND，想要"或者"要用 Q：

```python
from django.db.models import Q

# 价格低于 50 或者库存大于 100
CodeProduct.objects.filter(Q(price__lt=50) | Q(stock__gt=100))

# | 是"或"，& 是"且"，~ 是"非"
CodeProduct.objects.filter(Q(title__contains='源码') & ~Q(price__gt=100))
```

#### 冷知识：QuerySet 是"懒"的

```python
>>> qs = CodeProduct.objects.filter(price__lt=100)   # 这一刻并没有查数据库！
>>> qs = qs.filter(stock__gt=0)                      # 还没查！
>>> for p in qs:                                     # 真正用数据的那一刻才去查
...     print(p.title)
```

这叫**惰性求值**：你可以安心地拼装查询条件，Django 只在最后真正需要数据时才执行一次 SQL。

### 4.3 改（Update）

```python
# 方式一：查出来 → 改属性 → save()（适合改单个对象）
>>> p = CodeProduct.objects.get(id=1)
>>> p.price = 79.00
>>> p.stock = p.stock - 1        # 卖出一件，库存减 1
>>> p.save()                     # 老规矩：不 save 不落库

# 方式二：update() 批量更新（适合改一批，一条 SQL 搞定）
>>> CodeProduct.objects.filter(category='未分类').update(category='Web')
3    # 返回值：更新了 3 行

# 方式三：基于原值做运算（防并发错误的专业写法）
>>> from django.db.models import F
>>> CodeProduct.objects.filter(id=1).update(stock=F('stock') - 1)
# F('stock') 表示"数据库里 stock 当前的值"，由数据库完成减 1
# 如果两个顾客同时买最后一件，用方式一可能超卖，用 F 表达式就安全
```

**方式一 vs 方式二：**

| | 查出来改 + save() | update() |
|---|---|---|
| 执行 SQL | 2 条（SELECT + UPDATE） | 1 条（UPDATE） |
| 适合场景 | 改一个对象，还要用对象干别的 | 批量改一批数据 |
| 会触发 auto_now 刷新 | 会 | **不会**（直接发 SQL，不经过模型） |

### 4.4 删（Delete）

```python
# 删单个
>>> p = CodeProduct.objects.get(id=4)
>>> p.delete()
(1, {'home.CodeProduct': 1})    # 返回值：删了 1 行

# 批量删：先筛出来再删
>>> CodeProduct.objects.filter(stock=0, is_on_sale=False).delete()
```

⚠️ **delete 没有后悔药**，删了就真没了。生产项目里常做"软删除"——加一个 `is_deleted = models.BooleanField(default=False)` 字段，删除时只把它改成 True，查询时统一 `filter(is_deleted=False)`。数据还在，可恢复。

## 五、把 ORM 用到视图和页面里（串起来）

比如购物车页面展示"所有上架且有货的商品"：

**视图 `cart/views.py`：**

```python
from django.shortcuts import render
from home.models import CodeProduct

def index(request):
    products = CodeProduct.objects.filter(is_on_sale=True, stock__gt=0).order_by('-created_at')
    return render(request, 'cart/index.html', {'products': products})
```

**模板 `cart/templates/cart/index.html`：**

```html
<h1>代码商品列表</h1>
{% for p in products %}
    <div style="border:1px solid #ddd;margin:8px;padding:8px;">
        <h3>{{ p.title }} <small>¥{{ p.price }}</small></h3>
        <p>{{ p.description }}</p>
        <p>库存：{{ p.stock }} | 上架时间：{{ p.created_at|date:"Y-m-d" }}</p>
    </div>
{% empty %}
    <p>暂无商品</p>   <!-- 列表为空时显示这个 -->
{% endfor %}
```

模板语法三件套，记住就够用了：

- `{{ 变量 }}`：输出值
- `{% for %}` / `{% if %}`：逻辑标签
- `|date:"Y-m-d"`：过滤器（管道符，这里把日期格式化成 2026-07-21 这种样子）

## 六、进阶预告：表与表的关系（ForeignKey）

真实项目不可能只有一张表。比如"订单属于用户"：

```python
from django.contrib.auth.models import User

class Order(models.Model):
    user = models.ForeignKey(User, on_delete=models.CASCADE, verbose_name='买家')
    product = models.ForeignKey(CodeProduct, on_delete=models.CASCADE, verbose_name='商品')
    quantity = models.IntegerField('数量', default=1)
    created_at = models.DateTimeField(auto_now_add=True)
```

`ForeignKey` = 多对一（很多订单属于同一个用户）。`on_delete=models.CASCADE` 意思是：用户被删时，他的订单也一起删。

用起来像属性一样自然：

```python
>>> order = Order.objects.first()
>>> order.user.username          # 顺着订单摸到买家
'guozicheng'
>>> order.product.title          # 顺着订单摸到商品
'Django 实战源码'
>>> user.order_set.all()         # 反过来：某用户的所有订单
```

这块内容（一对多、多对多、关联查询的双下划线跨表语法 `order__user__username`）够再写一篇教程，先有个印象即可。

## 七、新手避坑清单（全文精华）

| # | 坑 | 典型现象 | 解法 |
|---|---|---|---|
| 1 | 改了 models.py 忘了迁移 | 查询报 `no such column: xxx` | 三步曲：改模型 → makemigrations → migrate |
| 2 | 新对象忘了 `save()` | 数据"神秘丢失" | create() 自带保存；手动 new 的必须 save() |
| 3 | `get()` 查不到，页面 500 | `DoesNotExist` 异常 | 用 `filter().first()` 或 try/except |
| 4 | 加非空字段被拦 | makemigrations 问你要 default | 提前写好 `default=...` 或 `null=True` |
| 5 | 双下划线写成一个 | `price_lt=100` 报 FieldError | 字段和条件之间是 `__`（两个下划线） |
| 6 | 循环里逐条 save 更新 | 几百条数据卡半天 | 批量用 `update()`、插入用 `bulk_create()` |
| 7 | 手动删 migrations 文件 | 迁移历史混乱，migrate 报错 | migrations 是档案，只增不删；玩坏按"黄金纪律第 4 条"重来 |
| 8 | null / blank 分不清 | 表单校验和数据库行为对不上 | null 管数据库，blank 管表单；字符串字段一般只加 blank |
| 9 | 金额用 FloatField | 对账差一分钱 | 金额永远 DecimalField |
| 10 | 表名和类名对不上 | 数据库里找不到 `CodeProduct` 表 | 表名 = `app名_类名小写`，如 `home_codeproduct` |

## 八、全文回顾（一屏看完）

```
【建表/改表 · 三步曲】
改 models.py  →  python manage.py makemigrations  →  python manage.py migrate

【增】
Model.objects.create(字段=值)          # 一步保存
obj = Model(字段=值); obj.save()       # 分两步
Model.objects.bulk_create([...])       # 批量

【查】
Model.objects.all()                     # 全部
Model.objects.filter(字段__条件=值)     # 筛选，返回列表，查不到是 []
Model.objects.get(id=1)                 # 查单个，查不到抛异常
.exclude() 排除 | .order_by('-字段') 排序 | [:N] 限量 | .count() 计数
多条件是 AND；要 OR 用 Q 对象

【改】
obj.字段 = 新值; obj.save()             # 改单个
Model.objects.filter(...).update(...)   # 批量改

【删】
obj.delete()                            # 删单个
Model.objects.filter(...).delete()      # 批量删
```

**三句话带走：**

1. 表结构的生命周期只有一条路：**改 models.py → makemigrations → migrate**，永远不要直接动数据库
2. 数据的增删改查全找"管家"：**`模型名.objects`** 点出来
3. 记牢三句保命口诀：**改完要 save，get 会报错，filter 返列表**
